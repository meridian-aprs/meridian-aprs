# Meridian APRS — Current State Verification Spec

**Purpose:** This document describes the expected state of the Meridian APRS application as of the completion of v0.1 Foundation, the UI Foundation milestone, and v0.2 Packets. A verification agent should use this spec to audit the codebase and report on what is present, what is missing, and what deviates from expectations.

**How to use this spec:** For each section, check whether the described capability or characteristic is present in the codebase. Report findings as: ✅ Present / ⚠️ Partial / ❌ Missing / 🔍 Unclear (needs runtime verification).

---

## Part 1 — Project Structure & Conventions

### Repository Layout
- The project is a Flutter application with a standard `lib/` source tree
- Core logic (packet parsing, transport) lives separately from UI code — they are not mixed together in the same files
- There is a `test/` directory with at least some test coverage for the packet parser
- Documentation files exist at: `CLAUDE.md`, `docs/ARCHITECTURE.md`, `docs/DECISIONS.md`, `docs/ROADMAP.md`, `docs/UI_UX_SPEC.md`

### Branching & Git
- There should be at least two feature branches open (`feat/ui-foundation`, `feat/v0.2-packets`) or their PRs should be visible
- No direct commits to `main` for feature work
- Commit messages do not contain AI attribution language ("made by Claude", "generated by", etc.)

### CI / Build
- GitHub Actions is configured for at minimum: build check, `dart format`, and `flutter analyze`
- The project builds without errors on at least one platform target

## Part 2 — v0.1 Foundation

### Map Display
- The app renders an interactive map using OpenStreetMap tiles
- The map is pannable and zoomable
- Station markers appear on the map representing real APRS stations received from the network

### APRS-IS Connectivity
- The app connects to the APRS-IS network (specifically `rotate.aprs2.net:14580` or equivalent)
- Connection state is tracked — the app knows whether it is connected or not
- The app can receive live APRS packets from the network

### Station Display
- Received stations are shown on the map at their reported coordinates
- Each station is represented by a marker (even if a placeholder icon at this stage)
- Tapping a station marker shows some information about that station (callsign at minimum)

### APRS Symbols (v0.1 scope)
- Station markers on the map reflect APRS symbol data from received packets
- The primary APRS symbol table (`/`) is referenced for icon selection
- Symbol rendering may be partial or simplified at this stage — full fidelity is a v0.2 concern

## Part 3 — UI Foundation Milestone

### Theme System
- The app supports three theme modes: Light, Dark, and Auto (system-adaptive)
- The default on first launch is Auto
- The user's theme preference is persisted and restored on next launch
- No hardcoded colors in the UI — all colors come from a centralized theme/token system
- The map tile style changes between light and dark modes (e.g. standard OSM tiles for light, dark tile set for dark)

### Adaptive Layout
- The app layout adapts based on screen width with at least two distinct breakpoints:
  - Narrow (phone): fullscreen map with floating action buttons and bottom sheets — no persistent side panel
  - Wide (tablet/desktop): a side panel or navigation rail becomes visible alongside the map
- The map remains visible and interactive in all layout modes — it is never fully replaced by another view
- Navigation between major sections (Map, Stations, Messages, Settings) works correctly in both layouts

### Core UI Widgets
The following UI components exist and are used in the app:

- **Connection status indicator** — a persistent, always-visible indicator showing whether the app is connected to a network or TNC. It is small and unobtrusive (not a modal or banner that blocks content). It conveys state with both color and a label or icon — not color alone.
- **Station list tile** — a reusable component for displaying a station entry in a list view, showing at minimum: callsign, APRS symbol, and last-heard time.
- **Beacon / transmit FAB** — a floating action button (or FAB cluster) for initiating a position beacon. Visible on mobile. Meets minimum tap target size (44×44pt).
- **Callsign input field** — a text field that validates amateur radio callsign format.
- **Bottom sheet component** — a reusable bottom sheet used for station detail, packet detail, or settings panels on mobile. It is draggable/dismissible.

### Settings Shell
- A Settings screen exists and is reachable from the main navigation
- Settings are organized into logical sections (connection, display, beaconing, etc.) — not a flat undifferentiated list
- Individual settings may be stubbed/non-functional at this stage, but the structure and sections are present

### Onboarding Flow
- A first-launch onboarding flow exists
- It is gated by a persistent flag — users who have completed onboarding do not see it again on subsequent launches
- The flow collects at minimum: callsign, SSID, and connection method preference
- There is a way to skip onboarding for experienced users
- Callsign input is validated before the user can proceed

## Part 4 — v0.2 Packets

### APRS Packet Parser
- A packet parser exists as pure Dart code with no native dependencies
- It accepts raw APRS string input (the APRS-IS text format)
- It returns a typed data model — not just a raw string or unstructured map
- The parser handles malformed or unknown packets gracefully (logs and skips — does not crash)

**Minimum packet types the parser supports:**
- Position (uncompressed, with and without timestamp)
- Position (compressed)
- Weather report
- Object
- Item
- Message
- Status report
- Mic-E

**Parser test coverage:**
- Unit tests exist for the packet parser
- Tests use real-world APRS packet strings as input fixtures (not synthetic/invented strings)
- Each supported packet type has at least one test case

### Packet Log UI
- A packet log view exists and is accessible from the main UI
- It displays a real-time scrolling list of received packets
- Each entry shows at minimum: timestamp, source callsign, packet type, and a human-readable summary
- Tapping a packet entry shows additional decoded detail
- The log is bounded — it does not grow unbounded in memory (a rolling buffer or cap is in place)
- Filtering by packet type is supported

### APRS Symbol Rendering
- APRS symbols are rendered using the correct symbol from the APRS symbol tables, not generic placeholder icons
- Both the primary table (`/`) and alternate/overlay table (`\`) are supported
- Overlay characters (alphanumeric overlays on overlay-capable symbols) are rendered correctly
- Symbols appear consistently in both the map view and the packet log
- Symbol rendering is abstracted behind a reusable widget — it is not ad-hoc inline logic repeated across the codebase

## Part 5 — UX Quality Expectations

These apply across all of the above milestones and should be assessed holistically.

### Visual Design
- The app does not look like a default Flutter demo app — it has a coherent visual identity
- Typography is consistent — font sizes and weights follow a clear hierarchy
- Spacing and padding are consistent — the layout does not feel ad-hoc
- The color palette is cohesive in both light and dark modes

### Interaction Quality
- All tappable/clickable elements meet a minimum 44×44pt touch target size
- State changes (connecting, receiving packets, theme switching) are reflected in the UI promptly — there are no stale displays
- Error states are communicated to the user — a failed connection attempt does not silently do nothing
- Loading states are handled — the app does not show empty/blank UI while waiting for data

### Accessibility Baseline
- Color is never the sole indicator of state — every status indicator pairs color with a label, icon, or both
- Interactive elements have meaningful labels for screen readers

### Map UX
- Tapping a station on the map produces a response (detail panel, bottom sheet, or callout) — it is not a dead tap
- Station markers do not overlap in a way that makes the map unusable at default zoom levels (basic clustering or z-ordering is acceptable)
- The map retains its last position and zoom level between sessions

## What is Explicitly Out of Scope at This Stage

The following features are **not** expected to be present yet. Do not flag their absence as a deficiency:

- KISS TNC over USB serial (v0.3)
- KISS TNC over BLE (v0.4)
- Position beaconing / transmit path (v0.5)
- Message sending (v0.5)
- Account sync or cloud features
- Web platform support
- GitHub Wiki or extended documentation
- APRS-IS filter builder UI (planned, not yet implemented)
- Push notifications
